## Context

`AExampleActorType` is declared as a normal script class in `Script/Game/Example_Actor.as`:

```angelscript
class AExampleActorType : AActor
```

The class does not explicitly specify `Blueprintable`, `NotBlueprintable`, `Abstract`, or `HideDropdown`. The Angelscript class generator currently gives non-statics script classes editor metadata `BlueprintType=true` and `IsBlueprintBase=true`, and the preprocessor supports explicit class specifiers that can later override those defaults.

The observed inconsistency is editor-entry specific:

- Angelscript's direct create-blueprint flow receives a resolved `UASClass` and creates the Blueprint directly from it.
- Unreal's standard Blueprint parent-class list goes through `UBlueprintFactory`, `SClassPickerDialog`, `SClassViewer`, `FBlueprintParentFilter`, and `FKismetEditorUtilities::CanCreateBlueprintOfClass`.
- UE's `CanCreateBlueprintOfClass` requires valid blueprint-base metadata and rejects deprecated/newer-version classes and most classes generated by another Blueprint unless derived-blueprint settings allow them.
- The parent-class picker also layers normal visibility and filter rules on top of that eligibility check.

This means "can create a Blueprint through Angelscript's direct popup" is not proof that "the standard parent-class list will display the same class." The fix should align those paths without weakening invalid-parent safeguards.

## Goals / Non-Goals

**Goals:**

- Make valid loaded `UASClass` types, including `AExampleActorType`, discoverable in the standard Blueprint parent-class picker.
- Preserve the existing Angelscript direct create-blueprint entry point.
- Keep explicit invalid-parent controls intact: `NotBlueprintable`, `Abstract`, `Deprecated`, and `HideDropdown` must still suppress classes.
- Add targeted diagnostics or tests so future regressions show which filter condition failed.

**Non-Goals:**

- Do not make every Angelscript class a Blueprint parent regardless of metadata.
- Do not bypass Unreal's standard Blueprint parent validation wholesale.
- Do not change Blueprint creation semantics for Blueprint-generated classes unrelated to Angelscript.
- Do not introduce a separate broad editor picker replacement unless the standard path cannot be integrated cleanly.

## Decisions

### Decision: diagnose first, then choose the narrow fix

The implementation should first capture the class state that the picker actually sees:

- `BlueprintType`
- `IsBlueprintBase`
- `CLASS_Abstract`
- `CLASS_Deprecated`
- `CLASS_HideDropDown`
- `CLASS_Hidden`
- `CLASS_NotPlaceable`
- `RF_Transient`
- `ClassGeneratedBy`
- `GetPackage()` / outermost package
- `FKismetEditorUtilities::CanCreateBlueprintOfClass`

Rationale: there are two plausible failure locations. If `CanCreateBlueprintOfClass(AExampleActorType)` is false, the fix belongs in generated class metadata/flags. If that returns true but the class still does not show in the list, the fix belongs in class-viewer discovery or filter integration.

Alternative considered: patch the editor picker first. That is riskier because it can hide a bad generated-class shape and create a second source of truth for parent eligibility.

### Decision: prefer metadata/flag correctness over picker-specific special cases

If the generated `UASClass` is missing or losing blueprint-base eligibility, the fix should be in `AngelscriptClassGenerator.cpp` or the preprocessor metadata flow. Valid script classes should look like valid Unreal blueprint bases to standard engine utilities.

Rationale: UE already uses `IsBlueprintBase` as the core contract. Keeping `UASClass` aligned with that contract lets the existing engine picker, reparent picker, and validation logic remain authoritative.

Alternative considered: add a custom Angelscript-only picker path. That would solve this one entry point but would not fix reparenting or other editor surfaces that ask UE whether a class is a valid Blueprint base.

### Decision: only adjust editor discovery if eligibility already passes

If `CanCreateBlueprintOfClass(AExampleActorType)` is already true, then the class data is correct and the missing list item is likely caused by class viewer collection, loaded/unloaded class handling, or additional filters. In that case, fix the narrow editor integration point so loaded `UASClass` objects are visible to the standard picker.

Rationale: this keeps runtime class generation unchanged and confines the patch to editor-only behavior.

Alternative considered: force explicit `Blueprintable` in `Example_Actor.as`. That would make the sample less useful as a default-behavior test and would not fix other valid script classes.

### Finding: eligibility passes; ClassViewer cache refresh is missing

The project-side diagnostic test confirmed that `AExampleActorType` is a loaded `UASClass`, derives from `AActor`, keeps `BlueprintType=true` and `IsBlueprintBase=true`, is not `Abstract`, `Deprecated`, `HideDropDown`, `Hidden`, or transient, passes `FKismetEditorUtilities::CanCreateBlueprintOfClass`, and passes the standard Blueprint parent class-viewer filter.

The remaining failure is therefore discovery freshness. UE's `SClassViewer` maintains a cached class hierarchy and only repopulates it when requested by editor events such as `GEditor->OnBlueprintCompiled()` or `GEditor->OnClassPackageLoadedOrUnloaded()`. Angelscript full reload already causes the Blueprint-compiled refresh path, but soft reload / initial-style materialization of a brand-new `UASClass` can create a valid loaded class without requesting the standard class-viewer hierarchy rebuild.

The fix should treat this as a cache invalidation problem:

- Runtime class generation broadcasts `GetOnClassReload(OldClass, NewClass)` for soft reload paths that materialize or replace loaded script classes, matching the full reload notification timing after class finalization.
- Editor reload integration marks that the class hierarchy changed when it observes an Angelscript class reload/create event.
- `OnPostReload` emits one `GEditor->BroadcastClassPackageLoadedOrUnloaded()` when the Angelscript class set changed, using Unreal's existing ClassViewer refresh hook rather than bypassing picker filters.

This preserves all eligibility checks and invalid-parent exclusions because it only asks the standard viewer to rebuild its cached hierarchy.

### Follow-up finding: editor module late startup also needs a refresh

During implementation, the repository already had a complete `AngelscriptEditor` module entry point and `FClassReloadHelper::Init()` wiring. The remaining gap was the late-startup case: if `AngelscriptEditor` starts after the Angelscript engine has already completed initial compilation, valid loaded `UASClass` objects may exist before the ClassViewer hierarchy has been asked to rebuild.

The editor module startup path should therefore mirror the existing component registry invalidation for an already-initialized engine:

- if `FAngelscriptEngine::IsInitialized()` and `IsInitialCompileFinished()` are both true, invalidate the component type registry as before;
- also request `GEditor->BroadcastClassPackageLoadedOrUnloaded()` so UE's standard ClassViewer refresh hook rebuilds the parent-picker hierarchy.

This is intentionally editor-only and does not change `UASClass` metadata, picker filters, or Blueprint parent eligibility.

### Decision: tests should exercise eligibility before UI clicks

The regression should target the same predicate chain the UI depends on, rather than requiring manual Slate interaction. A diagnostic test can assert that `AExampleActorType` is a valid Blueprint parent candidate and that explicitly hidden/invalid script classes remain rejected.

Rationale: UI automation for `SClassPickerDialog` is more fragile than checking the class flags, metadata, and filter predicates that drive it.

Alternative considered: rely on manual editor verification. Manual verification is still useful for final confirmation, but it does not protect against regression.

## Risks / Trade-offs

- Generated metadata may be correct, but the class viewer may still not enumerate `UASClass` objects from `/Script/Angelscript` → Mitigation: split diagnostics between `CanCreateBlueprintOfClass` and class-viewer filtering before implementing.
- Fixing discovery too broadly could expose transient or replaced hot-reload classes → Mitigation: preserve `CLASS_NewerVersionExists`, `CLASS_Deprecated`, `RF_Transient`, package, and `HideDropdown` filtering.
- Tests may need editor-only dependencies such as `UnrealEd` or `ClassViewer` → Mitigation: place coverage under editor/test modules that already depend on editor APIs.
- `AExampleActorType` has historical naming-conflict notes in technical debt docs → Mitigation: use path/package-aware diagnostics and avoid relying only on `FindFirstObject<UClass>("AExampleActorType")` if multiple loaded classes share the name.

## Migration Plan

1. Add diagnostics or a focused editor automation test to establish which predicate fails for `AExampleActorType`.
2. If `CanCreateBlueprintOfClass` fails, correct generated `UASClass` metadata/flags and add positive/negative coverage.
3. If `CanCreateBlueprintOfClass` passes, correct standard class viewer discovery/filter integration for loaded `UASClass` objects.
4. Manually confirm the editor workflow: create a Blueprint, search `AExampleActorType` in the parent-class list, and create/select it.
