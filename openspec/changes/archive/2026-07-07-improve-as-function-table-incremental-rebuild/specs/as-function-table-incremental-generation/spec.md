# Spec — as-function-table-incremental-generation

## ADDED Requirements

### Requirement: Unrelated Code Changes Do Not Trigger Full Function-Table Shard Recompilation

`AS_FunctionTable_*` shards generated by `AngelscriptUHTTool` SHALL recompile only when their corresponding reflection input actually changes. Source edits unrelated to bindings, such as `.cpp` edits without `UFUNCTION` reflection changes, SHALL NOT trigger full shard recompilation.

#### Scenario: Unrelated .cpp Edit Does Not Recompile All Shards

- **WHEN** a developer modifies a gameplay `.cpp` file with no reflection changes and runs an incremental build
- **THEN** `AS_FunctionTable_*` shards do not all recompile as a result, ideally with zero shard recompiles

#### Scenario: Reflection Changes Regenerate Correctly

- **WHEN** a module's `UFUNCTION` reflection definition changes
- **THEN** the affected corresponding shards regenerate and recompile, and generated artifacts remain deterministic and byte-stable for identical reflection input

### Requirement: Generated Artifacts Remain Equivalent After Refactoring

Incremental optimizations to the function-table generator, such as narrowing shard-injected includes or hardening incremental guards, SHALL NOT change generated binding semantics.

#### Scenario: Artifacts Are Equivalent Before And After Optimization

- **WHEN** codegen regenerates after an incremental-performance optimization
- **THEN** generated function-table binding logic remains equivalent to the pre-optimization output, with only include lines or recompilation granularity changing, and relevant binding tests pass
